# 大数据实战竞赛 - 模块二：离线数据处理教学

本模块基于 **Spark SQL / DataFrame API** 完成，所有代码位于 `com.ds.offline` 包下。由于操作涉及 Hive 存储与表结构变更，所有相关作业启动前需要确认 YARN / HDFS / Hive Metastore 服务正常。

## 1. 任务一：数据抽取 (`Task1_Extract.scala`)
### 需求分析
将 MySQL `ds_db01` 中的源表（`customer_inf`, `product_info`, `order_master`, `order_detail`）的增量数据，拉取到 Hive 的贴源层 (`ods`)。抽取策略是通过源表的 `modified_time` 字段过滤出大于 Hive 中现存最大时间戳的数据。最终将数据添加静态分区 `etl_date`（昨天的日期，格式 `yyyyMMdd`）。

### 核心代码讲解
*   **计算日期**：利用 Spark 内置函数 `date_format(date_sub(current_date(), 1), 'yyyyMMdd')` 获取昨天的日期字符串。
*   **JDBC读取**：通过 `spark.read.jdbc()` 加载 MySQL 全量数据。
*   **增量判断**：检查 Hive 中是否存在对应 `ods` 表，如果存在则执行 `max(modified_time)` 找出最新时间点，然后在 MySQL 加载到的 DataFrame 上执行 `filter(col("modified_time") > lit(maxModifiedTime))` 进行增量截取。
*   **写入 Hive**：设置 `hive.exec.dynamic.partition.mode=nonstrict` 开启分区支持，通过 `.mode(SaveMode.Append).partitionBy("etl_date").saveAsTable()` 将新数据挂载到对应分区。

### 执行命令
```bash
spark-submit --class com.ds.offline.Task1_Extract \
  --master yarn --deploy-mode client \
  target/bigdata-competition-1.0-SNAPSHOT-jar-with-dependencies.jar
```

## 2. 任务二：数据清洗 (`Task2_Clean.scala`)
### 需求分析
将 `ods` 库的增量分区数据抽取并清洗至 `dwd` 层明细宽表。要求：
1. 若只有“年月日”的字符串，转换为标准的 `timestamp`（补全 `00:00:00`）。
2. 将前一天 `ods` 的数据与 `dwd` 最新分区的数据做**全外连接 / union**，针对相同主键（如 `customer_id`），根据 `modified_time` 降序排序取第一条（即最新数据）。
3. 为每一行数据增加四个审计字段：`dwd_insert_user`, `dwd_insert_time`, `dwd_modify_user`, `dwd_modify_time`。新数据和修改后的数据，审计时间的维护规则不同（新增时 insert_time 取当前，修改时 insert_time 保持原值，modify_time 始终取当前）。
4. 过滤无效数据（例如城市长度大于 8 的订单）。

### 核心代码讲解
*   **时间补全**：编写工具函数 `formatTimestamp`，通过 `length(col) === 10` 判断并补全 `00:00:00`，再强转 `to_timestamp`。
*   **拉链 / 合并逻辑**：将 `ods` 与 `dwd` 历史数据通过 `unionByName` 合并，结合窗口函数 `Window.partitionBy("customer_id").orderBy($"modified_time".desc_nulls_last)` 给每组打上 `row_number()` 序号，只保留序号为 1 的记录。
*   **审计字段更新**：通过 `when($"dwd_insert_time".isNotNull, $"dwd_insert_time").otherwise(lit(currentTimeStr))` 判断该条数据是否为首次进入数仓，巧妙实现 `insert_time` 冻结与 `modify_time` 更新。

### 执行命令
```bash
spark-submit --class com.ds.offline.Task2_Clean \
  --master yarn --deploy-mode client \
  target/bigdata-competition-1.0-SNAPSHOT-jar-with-dependencies.jar
```

## 3. 任务三：指标计算 (`Task3_Metrics.scala`)
### 需求分析
此任务包含高达 10 道独立的 KPI 计算题。涉及表关联、复杂的窗口分析函数、中位数计算、行转列透视等。计算结果分别保存到 Hive 的 `dws` 汇总层，或直接通过 JDBC 写入到 ClickHouse (`shtd_result` 库) 中供展示。

### 核心计算逻辑解析
*   **用户与城市消费汇总 (dws)**: 关联订单事实表与客户维度表，按年/月/日分组，使用聚合函数 `sum` 和 `count` 统计金额与数量。其中，城市消费排行通过 `Window.partitionBy(省份, 年, 月).orderBy(总金额降序)` 计算分组排名 `sequence`。
*   **中位数比较 (ClickHouse)**: 中位数在 Spark 中使用内置表达式 `percentile_approx(order_money, 0.5)` 计算，随后将城市中位数与省份中位数进行 Join，通过 `when...otherwise` 得出“高/低/相同”。
*   **销售量/销售额双指标合并 (Top 10)**: 分别按 `product_cnt` 累加值和 `product_cnt * product_price` 累加值排序，由于各自要保留 1~10 的名次，使用 `row_number()` 获取排名作为关联键，将两个独立的排序结果横向拼接为一张结果表。
*   **连续两天下单用户占比**: 使用 `lag(日期, 1).over(Window.partitionBy(用户).orderBy(日期))` 函数，将用户前一次下单日期与当前下单日期放在同一行，通过 `datediff() === 1` 判定为连续购买，随后计算占比。
*   **省份累计订单量行转列**: 统计完各省总量后，将按照数量降序取出全省份名称数组，再传给 `.groupBy().pivot("province", 有序省份数组).sum("Amount")` 实现有序的行转列操作（Pivot）。
*   **5小时滑动窗口（步长1小时）**: 因为 Spark DataFrame 标准的窗口函数基于 Row 或 Range，处理 5小时 / 1小时 的滑动窗口时，通过转换为 Unix Timestamp，并在 Spark SQL 中利用 `JOIN` 自己（`t2.hour_ts BETWEEN t1.hour_ts - 4*3600 AND t1.hour_ts`）并配合 `HAVING COUNT(DISTINCT) = 5` 来确保时间窗口完整，实现复杂的滑动窗口聚合。

```bash

  --master yarn --deploy-mode client \
  target/bigdata-competition-1.0-SNAPSHOT-jar-with-dependencies.jar
```
